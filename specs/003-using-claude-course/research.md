# Research: Using Claude Course Specification

## Decision 1: Create a course specification and structure, not full lessons

**Decision**: Build GitBook-ready course reference pages, templates, chapter stubs, and navigation. Do not draft complete lesson content in this slice.

**Rationale**: The issue asks for a course specification, curriculum map, templates, prompt library, safety checklist, glossary structure, and folder structure. Full lesson drafting would be a separate larger content effort.

## Decision 2: Korean remains canonical

**Decision**: Korean is the source language. English and Japanese mirror file paths, structure, learning goals, examples, and safety requirements.

**Rationale**: This follows the constitution and keeps multilingual drift reviewable.

## Decision 3: Claude Code is an optional advanced path

**Decision**: Treat Claude Code as a conceptual advanced chapter, not a prerequisite for the main course.

**Rationale**: The target audience includes non-developers. Claude Code is useful for beginner developers and automation learners but should not make the course developer-dependent.

## Decision 4: Use plan-availability warnings instead of current feature claims

**Decision**: When mentioning projects, artifacts, file analysis, or Claude Code, state that availability can vary by plan and interface.

**Rationale**: Claude product surfaces can change. The course should teach durable thinking and verification habits rather than brittle feature instructions.
