# Codex CLI 101 강사용 진행 가이드

## 강의 목표

이 강의의 목표는 학습자가 Codex CLI 명령을 외우게 하는 것이 아니라, AI 코딩
에이전트와 안전하게 협업하는 기본 습관을 익히게 하는 것입니다. 수업이 끝난
학습자는 분석 요청, 수정 전 계획, 테스트 실행, 변경사항 리뷰를 스스로
구분할 수 있어야 합니다.

## 권장 진행 시간

- 90분 짧은 워크숍: 개념, 기본 사용, 변경사항 리뷰 중심
- 3시간 실습형 워크숍: 모든 장과 실습 일부 진행
- 1일 과정: 각 장의 실습을 팀별로 수행하고 리뷰 발표 포함

## 준비물

- 실습용 공개 저장소 또는 미리 준비한 장난감 프로젝트
- 민감한 정보가 없는 계정과 환경
- 터미널과 Git 기본 사용 가능 상태
- 테스트가 하나 이상 있는 예제 프로젝트

## 진행 흐름

1. 먼저 "Codex가 할 수 있는 일"보다 "사용자가 확인해야 할 일"을 강조합니다.
2. 설치 설명에서는 현재 공식 안내를 확인하게 하고, 오래된 설치 명령을 외우게
   하지 않습니다.
3. 첫 실습은 반드시 분석만 요청하게 합니다.
4. 버그 수정 실습에서는 Codex가 바로 파일을 바꾸지 않도록 계획 승인을 먼저
   받게 합니다.
5. 마지막에는 모든 학습자가 `git diff`를 읽고 리뷰 의견을 말하게 합니다.

## 토론 질문

- Codex에게 맡기면 편하지만 직접 확인해야 하는 일은 무엇인가요?
- 테스트가 통과해도 리뷰가 필요한 이유는 무엇인가요?
- 회사 저장소에서 Codex를 쓰기 전에 어떤 정책을 확인해야 하나요?
- 좋은 요청문과 나쁜 요청문은 결과를 어떻게 바꾸나요?

## 평가 기준

- 목표, 범위, 제한, 검증 방법이 들어간 요청문을 작성했다.
- Codex 설명과 실제 파일을 대조했다.
- 수정 전 계획을 받고 변경 범위를 좁혔다.
- 테스트 실패 로그의 핵심 원인을 설명했다.
- AI가 만든 변경사항을 리뷰 체크리스트로 검토했다.

## 운영 팁

초보자는 터미널 출력만으로도 부담을 느낄 수 있습니다. 먼저 "모르는 출력이
나오는 것은 정상"이라고 안내하고, Codex에게 로그를 설명하게 하는 과정을
보여주세요. 반대로 경험 있는 개발자는 AI에게 너무 넓은 권한을 주기 쉽습니다.
이 경우 작은 변경 단위와 리뷰 가능한 diff의 중요성을 강조하세요.

## 안전 안내

수업 중에는 실제 고객 데이터, 회사 기밀, API 키, 비밀번호, 개인 식별 정보가
들어 있는 저장소를 사용하지 않습니다. 학습자가 자신의 회사 저장소를 사용하고
싶어 한다면 먼저 조직 정책과 보안 규칙을 확인하게 하세요.
