# AI가 만든 변경사항 리뷰하기

## 오늘 배울 것

Codex가 만든 변경사항을 `git diff`로 확인하고, 의도, 범위, 위험, 테스트를
기준으로 리뷰하는 방법을 배웁니다.

## 왜 중요한가

AI가 만든 코드는 빠르게 보이지만 항상 팀의 의도와 맞지는 않습니다. 리뷰하지
않은 변경사항은 작은 버그보다 더 큰 신뢰 문제를 만들 수 있습니다.

## 쉬운 설명

리뷰는 "마음에 드는지"를 보는 일이 아닙니다. 다음 질문에 답하는 일입니다.

- 이 변경은 요청한 문제만 다루는가
- 새 동작이 설명 가능하고 테스트 가능한가
- 불필요한 리팩터링이나 파일 변경이 섞였는가
- 보안, 개인정보, 성능, 접근성 위험은 없는가
- 실패했을 때 되돌리기 쉬운가

Codex에게도 리뷰를 시킬 수 있지만, 최종 판단은 사용자가 해야 합니다.

## 따라 해보기

1. Codex가 만든 변경 후 `git diff`를 확인합니다.
2. Codex에게 diff를 요약하게 하되, "위험한 변경 후보"도 함께 찾게 합니다.
3. 변경 파일을 하나씩 열어 의도한 수정인지 확인합니다.
4. 필요하면 Codex에게 더 작은 패치로 줄이는 방법을 요청합니다.

## 흥미로운 문제

AI가 만든 변경 중 테스트와 무관한 포맷 변경이 많이 섞여 있다면 어떻게
처리해야 할까요? 리뷰하기 쉬운 변경 단위가 왜 중요한지 생각해 봅니다.

## Codex에게 물어볼 예제 프롬프트

```text
현재 git diff를 리뷰해줘.
요약, 의도한 변경, 의도하지 않았을 수 있는 변경, 테스트 필요 항목,
되돌리기 어려운 위험을 나눠서 알려줘.
```

## 자주 하는 실수

- Codex가 "완료"라고 말하면 그대로 믿는 것
- diff가 큰데도 한 번에 병합하는 것
- 테스트 파일 변경을 검토하지 않는 것
- 삭제된 코드의 의미를 확인하지 않는 것

## 체크리스트

- [ ] 변경 파일 목록을 확인했다.
- [ ] 요청 범위 밖의 변경이 있는지 봤다.
- [ ] 테스트와 수동 확인 결과를 함께 검토했다.
- [ ] 이해하지 못한 코드는 병합하지 않았다.
