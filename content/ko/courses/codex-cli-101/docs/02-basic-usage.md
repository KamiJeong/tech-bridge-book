# 터미널에서 Codex와 대화하기

## 오늘 배울 것

터미널에서 Codex에게 작업을 요청하는 기본 흐름과 좋은 요청문의 구조를
배웁니다.

## 왜 중요한가

Codex는 모호한 요청도 어느 정도 처리하지만, 좋은 결과는 좋은 맥락에서
나옵니다. 목표, 범위, 금지사항, 확인 방법을 함께 말하면 불필요한 수정과
오해를 줄일 수 있습니다.

## 쉬운 설명

좋은 Codex 요청은 작은 업무 지시서와 비슷합니다.

- 목표: 무엇을 알고 싶거나 고치고 싶은가
- 범위: 어떤 파일이나 기능을 봐야 하는가
- 제한: 아직 수정하지 말아야 하는가, 테스트만 실행해야 하는가
- 확인: 결과를 어떻게 검증할 것인가

처음에는 "분석만", "계획만", "수정 전 확인"처럼 단계별로 요청하는 것이
좋습니다.

## 따라 해보기

1. 실습용 저장소에서 Codex를 엽니다.
2. "이 프로젝트의 실행 방법을 찾아줘. 아직 명령은 실행하지 말고 후보만
   알려줘"라고 요청합니다.
3. Codex가 제안한 명령 중 하나를 선택해 실행 가능 여부를 물어봅니다.
4. 실행 결과를 보고 다음 질문을 이어갑니다.

## 흥미로운 문제

나쁜 프롬프트와 좋은 프롬프트를 비교해 봅니다.

- 나쁜 예: "이거 봐줘"
- 좋은 예: "로그인 테스트가 실패하는 이유를 찾아줘. 먼저 관련 파일과
  실행해야 할 테스트 명령을 정리하고, 아직 코드는 수정하지 마."

두 요청이 Codex의 행동을 어떻게 다르게 만들지 생각해 보세요.

## Codex에게 물어볼 예제 프롬프트

```text
이 작업을 바로 수정하지 말고 먼저 계획만 세워줘.
목표는 테스트 실패 원인 파악이고, 범위는 로그인 관련 파일이야.
필요한 파일, 확인할 가설, 실행할 명령을 순서대로 정리해줘.
```

## 자주 하는 실수

- 수정 허용 여부를 말하지 않는 것
- 테스트나 확인 방법 없이 결과만 받는 것
- 한 번에 여러 unrelated 작업을 시키는 것
- Codex가 물어본 clarifying question을 건너뛰는 것

## 체크리스트

- [ ] 목표를 한 문장으로 설명했다.
- [ ] 범위와 제한을 함께 말했다.
- [ ] 수정 전 계획을 요청할 수 있다.
- [ ] 검증 방법을 요청에 포함했다.
