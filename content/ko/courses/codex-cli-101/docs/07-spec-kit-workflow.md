# Spec Kit으로 요구사항에서 구현까지 가는 흐름

## 오늘 배울 것

Spec Kit을 사용해 막연한 요청을 사양, 계획, 작업 목록, 구현, 검증으로 나누는
흐름을 배웁니다.

## 왜 중요한가

AI 코딩 에이전트는 구체적인 지시가 있을수록 안전하게 일합니다. Spec Kit은
"무엇을 만들지"와 "어떻게 확인할지"를 먼저 정리하게 해 변경 범위와 리뷰
기준을 명확하게 만듭니다.

## 쉬운 설명

Spec Kit 흐름은 요리 전에 레시피를 쓰는 것과 비슷합니다.

1. Specify: 원하는 결과와 사용자를 설명합니다.
2. Plan: 어떤 구조와 방식으로 만들지 정합니다.
3. Tasks: 실제 작업을 작은 단위로 나눕니다.
4. Implement: 작업 목록에 따라 변경합니다.
5. Review: 결과가 사양과 맞는지 확인합니다.

Codex CLI와 함께 쓰면 "바로 구현"보다 "먼저 사양과 검증 기준"을 만들 수
있습니다.

## 따라 해보기

1. 작은 기능 요청을 한 문장으로 씁니다.
2. Codex에게 사용자, 목표, 완료 기준을 질문하게 합니다.
3. 사양 초안을 만든 뒤 빠진 위험이나 가정을 찾게 합니다.
4. 작업 목록을 만든 뒤 한 작업만 구현하게 합니다.

## 흥미로운 문제

"로그인 화면을 개선해줘"라는 요청은 너무 넓습니다. 이 요청을 Spec Kit에
맞게 바꾸려면 어떤 질문을 먼저 해야 할까요?

## Codex에게 물어볼 예제 프롬프트

```text
이 요청을 바로 구현하지 말고 Spec Kit 흐름으로 정리해줘.
먼저 사용자, 목표, 범위, 완료 기준, 위험한 가정을 질문해줘.
답을 받은 뒤 spec, plan, tasks 순서로 나눠줘.
```

## 자주 하는 실수

- 사양 없이 바로 구현하는 것
- 완료 기준을 "작동함"처럼 모호하게 쓰는 것
- 작업 목록이 너무 커서 리뷰하기 어려운 것
- 구현 후 사양과 결과를 다시 비교하지 않는 것

## 체크리스트

- [ ] 요청을 사용자와 목표 중심으로 설명했다.
- [ ] 완료 기준과 검증 방법을 적었다.
- [ ] 작업을 작은 단위로 나누었다.
- [ ] 구현 후 사양과 결과를 비교했다.
