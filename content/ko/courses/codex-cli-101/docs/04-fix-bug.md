# 버그 찾기와 수정 실습

## 오늘 배울 것

Codex와 함께 작은 버그를 찾고, 수정 계획을 세우고, 변경 범위를 검토하는
흐름을 배웁니다.

## 왜 중요한가

AI에게 "버그 고쳐줘"라고만 말하면 원인을 모른 채 코드가 바뀔 수 있습니다.
실무에서는 원인, 수정 이유, 검증 방법이 함께 남아야 안전합니다.

## 쉬운 설명

버그 수정은 네 단계로 나누면 다루기 쉽습니다.

1. 재현: 어떤 조건에서 문제가 생기는지 확인합니다.
2. 원인 가설: 어떤 파일이나 로직이 의심되는지 정리합니다.
3. 최소 수정: 문제와 직접 관련된 부분만 바꿉니다.
4. 검증: 테스트, 수동 확인, 코드 리뷰로 결과를 확인합니다.

Codex는 각 단계를 빠르게 도와줄 수 있지만, 수정 범위를 좁히는 판단은
사용자가 계속 확인해야 합니다.

## 따라 해보기

1. 실습 저장소에서 일부러 실패하는 작은 테스트나 오류 상황을 준비합니다.
2. Codex에게 실패 메시지를 요약하고 원인 후보를 세 가지로 좁혀 달라고
   요청합니다.
3. 수정 전 계획을 받은 뒤, 가장 작은 수정만 적용하게 합니다.
4. `git diff`를 읽고 의도하지 않은 변경이 있는지 확인합니다.

## 흥미로운 문제

Codex가 제안한 수정이 테스트는 통과하지만 코드가 더 복잡해졌다면 어떻게
할까요? "테스트 통과", "가독성", "변경 범위" 중 무엇을 더 확인해야 할지
정리해 보세요.

## Codex에게 물어볼 예제 프롬프트

```text
이 실패를 바로 고치기 전에 원인 후보를 먼저 정리해줘.
각 후보마다 확인할 파일, 확인할 방법, 예상되는 최소 수정 방향을 알려줘.
수정은 내가 승인하기 전까지 하지 마.
```

## 자주 하는 실수

- 실패 재현 없이 바로 수정하는 것
- 여러 버그를 한 번에 고치게 하는 것
- 원인 설명 없이 패치만 받는 것
- `git diff`를 읽지 않고 테스트 결과만 보는 것

## 체크리스트

- [ ] 버그를 재현하는 방법을 적었다.
- [ ] 원인 후보를 확인한 뒤 수정했다.
- [ ] 변경 범위가 작고 설명 가능하다.
- [ ] 테스트와 diff를 모두 확인했다.
