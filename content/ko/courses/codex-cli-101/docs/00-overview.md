# Codex CLI 개념 소개

## 오늘 배울 것

Codex CLI가 무엇인지, 일반 챗봇과 무엇이 다른지, 터미널에서 AI 코딩
에이전트를 사용할 때 어떤 태도가 필요한지 배웁니다.

## 왜 중요한가

Codex CLI는 단순히 답을 말해주는 도구가 아니라 저장소를 읽고, 파일을
수정하고, 명령을 실행하며, 변경사항을 설명할 수 있는 작업 파트너입니다.
그래서 처음부터 "무엇을 맡기고 무엇을 직접 확인할지"를 구분해야 합니다.

## 쉬운 설명

일반 챗봇에게 코드를 물어보면 대화창 안에서 설명을 받습니다. Codex CLI는
터미널 안에서 프로젝트 폴더를 함께 보며 일합니다. 예를 들어 "이 오류의
원인을 찾아줘"라고 말하면 관련 파일을 검색하고, 필요하면 테스트를 실행하고,
수정 후보를 제안할 수 있습니다.

하지만 Codex가 프로젝트를 볼 수 있다고 해서 항상 맞는 판단을 하는 것은
아닙니다. Codex는 맥락을 추론하지만 제품 목표, 팀 규칙, 보안 정책, 배포
위험을 완전히 알지는 못합니다.

## 따라 해보기

1. 실습용 저장소를 하나 엽니다.
2. 터미널에서 프로젝트 루트로 이동합니다.
3. Codex에게 "이 저장소가 어떤 구조인지 먼저 설명해줘. 아직 파일은 수정하지
   마."라고 요청합니다.
4. 답변에서 이해되지 않는 폴더 이름을 하나 고르고 다시 설명을 요청합니다.

## 흥미로운 문제

Codex에게 "이 프로젝트에서 초보자가 가장 먼저 읽으면 좋은 파일 5개를
추천하고 이유를 설명해줘"라고 물어보세요. 추천 결과가 실제 파일 구조와
맞는지 직접 확인해 봅니다.

## Codex에게 물어볼 예제 프롬프트

```text
이 저장소를 처음 보는 개발자라고 생각하고 설명해줘.
아직 파일을 수정하지 말고, 주요 폴더와 실행 흐름만 정리해줘.
확실하지 않은 부분은 추측이라고 표시해줘.
```

## 자주 하는 실수

- 처음부터 "다 고쳐줘"라고 요청해서 변경 범위가 커지는 것
- Codex 설명을 읽지 않고 바로 수정 결과만 보는 것
- 비밀 정보가 들어 있는 파일을 그대로 분석 대상으로 넘기는 것
- 프로젝트 규칙을 설명하지 않고 일반적인 방식으로 고치게 하는 것

## 체크리스트

- [ ] Codex CLI를 챗봇이 아니라 작업 에이전트로 이해했다.
- [ ] 파일 수정 전 분석만 요청하는 방법을 안다.
- [ ] Codex 답변에서 추측과 확인된 사실을 구분한다.
- [ ] 민감한 정보를 프롬프트에 넣지 않는다.
