# 테스트 실행과 실패 원인 분석

## 오늘 배울 것

Codex에게 테스트 명령을 찾게 하고, 실패 로그를 해석하고, 다음 확인 단계를
정하는 방법을 배웁니다.

## 왜 중요한가

테스트는 AI가 만든 변경사항을 확인하는 가장 기본적인 안전장치입니다. 하지만
테스트가 실패했을 때 로그를 읽지 못하면 Codex가 제안하는 다음 행동을 평가할
수 없습니다.

## 쉬운 설명

테스트 실패 로그는 보통 세 부분으로 나눠 읽습니다.

- 어떤 테스트가 실패했는가
- 기대값과 실제값이 어떻게 다른가
- 실패가 코드 변경 때문인지 환경 문제인지

Codex에게 로그 전체를 던지기보다 "핵심 실패 줄을 요약하고 다음 확인 명령을
제안해줘"라고 요청하면 더 좋은 답을 얻기 쉽습니다.

## 따라 해보기

1. 저장소의 테스트 명령을 Codex에게 찾게 합니다.
2. 테스트를 실행하기 전에 어떤 시간이 걸릴지, 어떤 범위를 확인하는지 묻습니다.
3. 실패하면 로그에서 첫 번째 실패 지점을 요약하게 합니다.
4. Codex가 제안한 다음 명령을 실행하기 전에 이유를 설명하게 합니다.

## 흥미로운 문제

테스트 실패가 내 코드 때문인지, 기존에 깨져 있던 테스트인지 어떻게 구분할 수
있을까요? 실행 순서, 브랜치 상태, 최근 변경 파일을 기준으로 생각해 보세요.

## Codex에게 물어볼 예제 프롬프트

```text
이 저장소에서 실행할 수 있는 테스트 명령을 찾아줘.
명령을 바로 실행하지 말고, 각 명령이 확인하는 범위와 예상 소요 시간을
먼저 알려줘. 빠른 확인용 명령을 우선 추천해줘.
```

## 자주 하는 실수

- 가장 무거운 전체 테스트부터 실행하는 것
- 실패 로그의 첫 원인을 건너뛰는 것
- 환경 문제와 코드 문제를 구분하지 않는 것
- 테스트 통과 후에도 변경사항 리뷰를 생략하는 것

## 체크리스트

- [ ] 테스트 명령의 목적을 이해했다.
- [ ] 실패 로그의 첫 번째 의미 있는 오류를 찾았다.
- [ ] 다음 확인 명령을 실행하기 전에 이유를 확인했다.
- [ ] 테스트 결과와 코드 diff를 함께 검토했다.
